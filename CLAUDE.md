## What this repo is

"Reeled In" — a Hack the 6ix 2026 (Toronto, Jul 17–19, 36h) project: neural A/B testing for short-form video using Meta's TRIBE v2 brain-encoding model. Users upload video variants (or a voiceover script → ElevenLabs-generated voice variants); the app predicts 5-network brain engagement time-series and declares a winner.

**Current state: planning docs only — no code yet.** Code lands under `backend/`, `frontend/`, and `demo/` per the repo layout in `PARALLEL_IMPLEMENTATION_PLAN.md`. There are no build/test/lint commands until then.

## Contract essentials (from CONTRACTS.md — read it before writing any code)

- **snake_case everywhere** — wire JSON, Mongo fields, Python, frontend keys. No camelCase, no translation layer.
- IDs are prefixed strings: `usr_`, `test_`, `var_`, `score_` + 12-char lowercase hex, generated by the backend. The test identifier is `test_id` everywhere — never `id`/`testId`/`run_id`.
- The 5 networks (fixed order, exact keys): `visual`, `auditory`, `language`, `motion`, `default_mode`. The 4 metrics: `peak`, `sustained`, `retention`, `overall` — floats in [0,1]. Test types: `upload`, `voice`. Statuses: `pending`, `scoring`, `complete`, `failed`.
- The **Score Object** (§3) is the central shape: B produces it, C stores it (Mongo `scores`), A renders it. Each `networks.<name>` array has length `round(duration_sec * sample_rate_hz)` at 1 Hz, already normalized — the frontend never re-normalizes. Failed score → `null` score + test `status: "failed"`.
- API is under `/api` (routes in §5, locked once the frontend codes against them); errors use `{ "error": { "code", "message" } }`; auth is `Authorization: Bearer <auth0_jwt>` (client-side Auth0 SPA SDK; the Base44 app is Public), verified by the backend against Auth0's JWKS → `user_id`, with CORS locking the origin (see §6).
- Mongo: DB `reeled_in`, plural snake_case collections (`users`, `tests`, `variants`, `scores`), `_id` holds the prefixed string ID. Media is referenced only by opaque `media_key` (`media/<variant_id>.mp4`).
- Env var names are fixed in §8 (`MONGODB_URI`, `HF_TOKEN`, `ELEVENLABS_API_KEY`, `BACKBOARD_API_KEY`, `MODAL_TOKEN_ID/SECRET`, `ALLOWED_ORIGINS`, `AUTH0_DOMAIN`, `AUTH0_AUDIENCE`, optional `GEMINI_API_KEY`).

## Architecture

Two "backends" by design: **Base44** hosts the frontend (a Public app; user login is a client-side **Auth0** SPA SDK, not Base44 native — see CONTRACTS §6), and can't run heavy ML Python or reach Atlas reliably; **FastAPI on Modal** is the real backend — all logic, secrets, and orchestration. The FastAPI app calls a Modal A100 GPU function for TRIBE v2 scoring (demo scores precomputed and seeded into Mongo), ElevenLabs + ffmpeg for voice-variant generation, Backboard for LLM suggestions/memory, and MongoDB Atlas (pymongo only) as the system of record.

Planned `backend/` structure (owners in brackets, per the four-lane split A=frontend, B=scoring, C=backend/data, D=generation):
- `main.py` + `modal_app.py` — FastAPI served as a Modal ASGI endpoint [C]
- `models/schemas.py` — Pydantic models mirroring CONTRACTS [C]
- `api/routes_*.py` — the §5 routes [C]
- `db/mongo.py`, `db/seed_demo.py` — pymongo + demo-score seeding [C]
- `scoring/` — TRIBE v2 load, 20k-vertex → 5-network reduction, metrics; public entrypoint `score(media) -> ScoreObject` [B]
- `generation/` — ElevenLabs voice reads, ffmpeg audio overlay, `generate_voice_variants()`, Backboard LLM [D]
- `mocks/` — canned ScoreObject/variants so all lanes build independently until Phase 3 integration

Build order: Phase 0 freeze contracts → Phase 1 everyone builds against mocks (max parallelism) → Phase 2 real pipelines → Phase 3 integration (B→C and D→C, then C→A, then seed demo scores) → Phase 4 demo hardening. B's TRIBE `score()` is the critical path.

## Project constraints worth remembering

- TRIBE v2 is **CC BY-NC 4.0** — hackathon demo fine, no paid product (hence free-beta/waitlist framing).
- TRIBE needs the gated HuggingFace LLaMA-3.2 license accepted before any run (`HF_TOKEN`).
- fMRI output is ~1 Hz and BOLD-lagged — metrics are meaningful at ~5 s granularity, not per-frame; the pitch deliberately frames scores as a directional engagement proxy, not ground truth.
- The demo path relies on **precomputed** scores seeded into Mongo; live GPU scoring is the technical core but the demo must not depend on it succeeding live.
